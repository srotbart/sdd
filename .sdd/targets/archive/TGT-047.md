# Target: Live agent data pipeline — name field in register message, server→client flow, replace MOCK_AGENTS

## Current statement

The hub has no live agent data path from server to UI. Every screen that renders agent names or counts uses `MOCK_AGENTS` — a hard-coded constant in `App.tsx:127-129` with two fake entries. This is confirmed at every consumer: Dashboard (`App.tsx:410`), Session (`App.tsx:419`), WorkItems (`App.tsx:427`), Activity (`App.tsx:429`), Header (`App.tsx:441`).

Three gaps compose this failure:

**1. No `name` field in the agent register message.**
`ws-agent.ts:9-14` defines `RegisterMessage { type, workspace, pid, host }` — no `name` field. The DB agents table (`schema.ts:11-22`) also has no `name` column. The client `Agent` type (`types.ts:68-75`) requires `{ id, initials, name, host, status, pid }` but the server never provides `name` or `initials`. The register message must be extended to accept an optional `name` field (or derive initials server-side from `pid@host`).

**2. Client drops agent state from WebSocket.**
Even if the server sent agents with names, `App.tsx:294` returns early for any message type that isn't `sdd-changed`, so snapshot and update messages carrying `getAllAgents()` data are silently discarded (depends on TGT-046 for the WebSocket consumption fix).

**3. No `liveAgents` state in the client.**
`App.tsx` has no `liveAgents` state variable. All components receiving agents get `MOCK_AGENTS`. Once the WebSocket pipeline is fixed, the client needs a `liveAgents: Agent[]` state populated from snapshot/update messages and wired to all screens.

The spec must define: how agents provide a human-readable name (agent self-reports via register message, or server derives from `pid@host`), how `initials` are computed, and that `MOCK_AGENTS` is replaced by live state.
