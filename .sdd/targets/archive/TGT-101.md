# Target: Enforce user-authored coding standards in the review/refactor phase

## Current statement

Repositories using SDD (those with a `.sdd/` directory) should have **user-authored
coding standards** that Claude follows and that are enforced in the **review/refactor
phase** of the pipeline — keeping the spec and `spec-audit` focused purely on behaviour
while conventions, style, and best practices are handled where refactoring happens.

**Framing — SDD as TDD for agents.** The standards are the "clean conventions and
patterns" you refactor *into*, i.e. the refactor leg of the cycle:

| TDD | SDD |
|---|---|
| test (red) | spec |
| mvp code (green) | code |
| refactor → clean conventions/patterns | review → gaps → refactor (work items) |

So standards belong to **review/refactor**, not to spec. `spec-audit` and the spec remain
about *what must be true* (behaviour); standards are a separate concern applied during
review and cleanup.

1. **Standards are a non-spec artifact, user-authored.** The user is the source of truth;
   the agent does not invent or infer them. They are **not** a spec domain and are **not**
   swept by the normal `spec-audit` (which stays focused on the spec). Scope spans
   formatting/style, best practices / anti-patterns, project-specific conventions, and
   architectural rules.

2. **Enforced across three layers (no spec-audit layer).**
   - **Proactive** — Claude reads the standards and follows them *as it writes*, so the
     "green" code already trends toward the conventions (violations prevented at source).
   - **Active blocking** — linters/formatters plus a hook or CI gate that rejects code
     violating the mechanically-checkable rules.
   - **Review-time (the refactor leg)** — the TGT-096 reviewer team audits code against
     the standards and emits issue artifacts for the judgement-based rules linters can't
     express; those become **refactor work items**. This is the agent analogue of TDD's
     refactor step.

3. **Applies wherever `.sdd/` exists** — an SDD capability that activates in any repo
   using SDD, not a one-off for this project.

Relationship: the standards are the **rubric** for the TGT-096 reviewer; this target is
effectively the "what we refactor into" half of the review/refactor phase and should be
designed alongside TGT-096 so review and standards share one definition of "a violation."

Open decisions to settle at target-engage (not pinned here):
- **Where standards live** — a dedicated `.sdd/standards/` artifact, CLAUDE.md + tooling
  config, or both (CLAUDE.md for proactive delivery, a structured file for the review
  rubric). Explicitly *not* a spec domain.
- **Proactive delivery mechanism** — context injection vs. SessionStart hook vs.
  PreToolUse hook on Edit/Write; how standards reach Claude reliably across sessions.
- **Active-blocking wiring** — which linters/formatters per language, pre-commit hook vs.
  CI gate, and how rules that aren't lint-expressible are routed to the reviewer instead.
- **Deterministic vs. judgement split** — formatting/style is lint-checkable; anti-patterns
  and architectural rules need LLM judgement (the reviewer). Make the split explicit.

Scope note: a `workflow`/plugin capability in the review/refactor phase; integrates with
TGT-096 and the work-item flow. Standards content itself is user-authored data, not spec.
Exact decomposition settled during target-engage.
