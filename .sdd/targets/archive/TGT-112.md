# Target: Comment on selected text in the Projection view

## Current statement

In the Projection view, the user can mark (select) text and attach a comment to it, which is
persisted next to the projection and later addressed (and pruned) by the agent.

1. **Select → marker icon** — when the user selects/marks text in the projection, a small
   icon appears next to the selected text.
2. **Click icon → action menu** — clicking the icon shows these options:
   **clarify**, **re-evaluate**, **expand**, **condense** — with an optional small
   multiline input (a hovering text box) to write a note alongside the chosen action.
3. **Confirm → persist as JSON** — on a confirm action, a JSON file is created/updated **at
   the same location as the projection file**. A new entry is appended containing: the
   **selected text**, the **line**, the **action**, and the user's **input**.
4. **Highlight + type label** — commented text is highlighted, with a **tiny label** (small
   text ~half size, aligned at the end to the top of the last word of the highlighted text)
   indicating which comment type it is.
5. **Hover detail** — hovering over the commented line(s) shows the **date and time** the
   comment was added, including the comment text.
6. **Lifecycle** — once the agent addresses a comment, that entry is **removed** from the
   JSON file.

These behaviors (the four action types, optional multiline note, JSON co-located with the
projection, the entry fields, the tiny end-aligned type label, the hover date/time+comment,
and removal-on-address) are the user's requirements and are pinned.

Resolved decisions (see Dialog 2026-06-05 — Agent) — decomposed into three spec items:
- **JSON filename & schema** — co-located `.sdd/projections/<name>.comments.json`; an array of
  entries `{ id, action, selectedText, line, note, createdAt }`.
- **Selection → source mapping** — the selected text is located in the raw markdown source the
  client already holds; the 1-based line of the first occurrence is recorded.
- **Multiple / overlapping comments** — additive; the JSON is an array, highlights stack.
- **Server read/write** — new Hub endpoints `GET` / `PUT` / `DELETE`
  `/workspaces/:id/projections/:name/comments`, path-sanitized (architecture domain).
- **"Agent addresses it"** — a NEW dedicated SDD skill reads the comments JSON, performs the
  action on the projection, and prunes addressed entries (workflow domain) — **built in this
  pass** (per user).

Decomposition (per user — split across domains):
- **SPEC-scr-053** (ui-screens) — the Projection comment UI.
- **SPEC-arch-042** (architecture) — Hub server read/write/delete of the co-located JSON.
- **SPEC-wf-034** (workflow) — the skill that addresses comments and prunes handled entries.
