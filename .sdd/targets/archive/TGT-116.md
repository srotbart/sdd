# Target: Back / forward buttons navigate between viewed content

## Current statement

The app shell shows browser-style **back** and **forward** buttons that navigate through the
history of viewed content — screens (tabs) and the content viewed within them (selected
artifacts, the plugin reference, the active workspace). Going back restores the previously
viewed view state; going forward re-applies one that was backed out of.

1. **Back / forward buttons** in the app shell (header) navigate through view history.
2. **What is a "view"** — the existing view state already tracked and URL-synced in `App.tsx`:
   workspace + tab (`activeTab`) + selected artifact (`selectedItemId`) + plugin-reference
   (`pluginRefActive`). Each distinct view the user lands on is a history entry.
3. **Disabled at the ends** — back is disabled at the oldest entry, forward at the newest.
4. **Forward truncation** — navigating to a new view after going back discards the forward
   entries (standard browser behavior).
5. **State applied on navigate** — moving back/forward restores the full view state (tab,
   selected item, plugin-ref, workspace) and keeps the URL in sync.

Resolved decisions (see Dialog 2026-06-06 — Agent):
- **Source of truth:** an in-app navigation stack (array of view states + a current index) in
  `App.tsx` — needed for correct disabled states, since the History API exposes no `canGoBack`.
  The existing URL sync (`pushUrlState` / `replaceState`) is kept so reload/shareable links work.
- **View state shape:** `{ workspaceId, tab (activeTab), selectedItemId, pluginRefActive }`
  (the tuple already URL-synced). Consecutive duplicate states are not pushed.
- **Surfaces:** on-screen back/forward buttons in the app-shell header, plus `Alt+←` / `Alt+→`
  keyboard shortcuts. Hijacking the browser's native back button and mouse aux-buttons is a
  deliberate non-goal for this pass (the buttons drive an in-app stack, not browser history).
- **Single domain:** `ui-layout` — the buttons are shell chrome and the stack lives in `App.tsx`.

## Dialog

### 2026-06-06 — User
Back and forward buttons should navigate between viewed content — be it screens, content like
artifacts, etc.

### 2026-06-06 — Agent
The intent is clear and the app already has the foundation: `App.tsx` tracks view state as
`{ activeWorkspaceId, activeTab, selectedItemId, pluginRefActive }` and syncs it to the URL —
but via `history.replaceState` (no history entries) with no `popstate` handler, so back/forward
does nothing today. Resolving the design from that:

- **In-app stack, not raw History API.** Maintain `navHistory: ViewState[]` + `navIndex` in
  `App`. A user navigation truncates any forward entries, appends the new state, advances the
  index; back/forward move the index and re-apply the stored state without pushing a new entry
  (guard flag). This gives precise `canGoBack`/`canGoForward` for disabled buttons — which the
  bare History API cannot, since it has no `canGoBack`. URL stays synced via the existing
  `replaceState` path for reload/share.
- **Scope:** on-screen header buttons + `Alt+←` / `Alt+→`. Browser-native back button and mouse
  back/forward (aux) buttons are out of scope here to avoid fighting the browser's own history;
  can be a follow-up that layers `pushState` + `popstate` on top of the stack.
