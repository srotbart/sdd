# Target: sdd-worker agent and skill for execution-phase handoff

## Current statement
The SDD workflow splits into two phases:

1. **Intent phase** (human + Claude): creating targets, engaging them via `/sdd:target-engage`, and reconciling with the spec. This phase requires judgment and human sign-off.
2. **Execution phase** (sdd-worker agent): spec audit, gap creation, work item decomposition, and work item closure. This phase is deterministic and runs autonomously.

A skill `/sdd:spawn-sdd-worker` invokes the Agent tool to spawn a persistent agent named `sdd-worker` with instructions to run the full execution pipeline (`sdd:spec-audit` → `sdd:gap-to-work-items` → `sdd:work-item-close` loop) for a specified domain. The worker is not a subagent — it is created with the intention of being messaged and reused across pipeline runs.

`sdd:session-start` next-action footer: when open gaps or pending/blocked work items exist, suggest `/sdd:spawn-sdd-worker {domain}` instead of the individual pipeline skills.

`sdd:init` mentions the two-phase model: once the spec is established, the user spawns `sdd-worker` to handle execution autonomously.
