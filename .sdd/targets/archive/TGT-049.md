# Target: CommandPalette navigation selects the item and URL deep-links restore item selection

## Current statement

Two related gaps in navigation/selection:

**1. CommandPalette discards the item ID (bug).**
`App.tsx:477`: `onNavigate={(kind, _id) => {` — the `_id` parameter (underscore prefix = intentionally unused) is never passed to `setSelectedItemId`. The handler only calls `setActiveTab(tab)`, switching the visible screen but not selecting the specific artifact. Selecting "GAP-arch-001" in the palette opens the Gaps tab but no gap is highlighted or scrolled to. SPEC-scr-019 says "Enter navigates and closes" — the intent is item-level navigation, not tab-level navigation. Fix: call `setSelectedItemId(_id)` after `setActiveTab(tab)`.

**2. URL-encoded `&id=` is never used to restore item selection on mount.**
`App.tsx:247-250` reads `selectedItemId` from URL params on mount. `App.tsx:271` writes it back to the URL on navigation. But none of the screens receive the initial `selectedItemId`: `Targets.tsx` is rendered at `App.tsx:421` with no `initialTargetId`, `Gaps.tsx` at line 425 with no `initialGapId`, `WorkItems.tsx` at line 427 with no `initialWiId`. Each screen accepts these props (they are in the component signatures) but they are never passed. After a page refresh with `?v=gaps&id=GAP-arch-001` in the URL, the Gaps tab opens but no gap is selected. SPEC-scr-023 requires refresh to restore the same view.

The spec must explicitly state: (a) `onNavigate` in the CommandPalette callback must call `setSelectedItemId` with the provided id, and (b) each screen's `initialXxxId` prop must be wired from the URL-parsed `selectedItemId` on mount.
