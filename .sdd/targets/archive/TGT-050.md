# Target: POST /workspaces server spec, duplicate-path handling, and watcher lifecycle

## Current statement

Three related architecture gaps around workspace CRUD:

**1. POST /workspaces has no spec item.**
`server/index.ts:171-203` implements `POST /workspaces` accepting `{ name: string, path: string }`. SPEC-arch-010 covers `GET /workspaces`, SPEC-arch-011 covers `PATCH /workspaces/:id`, but there is no spec item for the POST endpoint. Missing spec coverage: required fields, 400 validation shape, 201 response shape, and the side-effect of starting a file watcher for the new workspace path.

**2. Duplicate-path POST silently returns 201 with null body.**
`db/index.ts:40` uses `INSERT OR IGNORE`. If the same `path` is submitted twice (path has a UNIQUE constraint per `schema.ts:5`), the INSERT is silently skipped, `getWorkspaceById(id)` is called with a new UUID that was never inserted, returns `undefined`, and the server responds with HTTP 201 and a null body. `AttachWorkspaceDialog.tsx:79` calls `onAttached()` unconditionally — the UI behaves as if the attach succeeded. The user gets no feedback. No spec item defines this scenario. Fix: the server should return 409 (Conflict) when the path already exists and the client should surface the error.

**3. Watcher cleanup functions are discarded — watcher lifecycle is unspecced.**
`startWatcher` returns a cleanup `() => void` function (`watcher.ts:97-102`). The POST handler (`index.ts:190`) and `startWatchers` (`index.ts:256-273`) both discard the return value. If a workspace's path is changed via PATCH, the old chokidar watcher continues watching the stale path indefinitely and no new watcher is started for the updated path. Similarly, if the server ever removes a workspace, the watcher is never stopped. No spec item covers: watcher teardown, PATCH-triggered watcher restart, or startup cleanup of stale agent rows.

The spec must define: the POST /workspaces request/response contract and error codes; 409 behavior on duplicate path; and that PATCH /workspaces/:id must stop the old watcher and start a new one when `path` changes.
