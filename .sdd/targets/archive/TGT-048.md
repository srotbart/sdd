# Target: Dashboard and Activity screens receive live data instead of mocks

## Current statement

Both the Dashboard and Activity screens are permanently connected to hard-coded mock data:

**Dashboard (`SPEC-scr-001`)**: `App.tsx:408-414` passes `MOCK_WORKSPACES`, `MOCK_AGENTS`, and `MOCK_ACTIVITY` to `<Dashboard>`. The `MOCK_WORKSPACES` shape is `WorkspaceData[]` which includes `counts`, `agents[]`, and `lastActivity` fields. The live REST endpoint `GET /workspaces` returns only `DbWorkspace[]` — `{ id, name, path, description, created_at }` — with none of those fields. There is no spec item defining how the Dashboard gets live `WorkspaceData` (with counts of targets/gaps/work items per workspace, connected agents, and last activity timestamp).

**Activity screen (`SPEC-scr-007`)**: `App.tsx:429` passes `MOCK_ACTIVITY` and `MOCK_AGENTS`. The server emits `{ type: "activity", agentId, event, detail }` WebSocket messages from `ws-agent.ts:92-97` when agents emit events, but the client's WebSocket handler (`App.tsx:289-319`) never processes `type: "activity"` messages. There is no spec item defining the activity event shape, how the client accumulates activity lines, or what the line count cap is.

Spec items needed:

1. **Dashboard workspace data source**: `GET /workspaces` should return enriched `WorkspaceData` (or a new `GET /workspaces/dashboard` endpoint should exist) including live counts of active targets/gaps/work-items per workspace, connected agent IDs, and last-activity timestamp. A spec item must define this response shape and how Dashboard consumes it.

2. **Activity event pipeline**: a spec item defining the `activity` WebSocket message shape (`{ type: "activity", agentId: string, workspaceId: string, kind: "in" | "note" | "err", msg: string, t: string }`), that the client accumulates these into an `ActivityLine[]` capped at N lines (e.g. 500), and wires the live list to the Activity screen.
