# Target: sdd-help lists and explains every skill, kept current and drift-free

## Current statement

The `sdd-help` skill must list **and** provide help on **all** skills in the plugin — every
skill present under `plugin/skills/` appears in `sdd-help` with usable help content (what it
does and how/when to invoke it). This coverage must stay up to date as skills are added,
removed, renamed, or re-described, so `sdd-help` never drifts from the actual skill set.

- `sdd-help` is **complete**: no skill is missing from it, and it lists no skill that no
  longer exists.
- `sdd-help` is **informative**: each listed skill has help content, not just a name.
- `sdd-help` is **current**: drift between `sdd-help` and the real skills is detected and
  prevented from shipping, the same way other skill-documentation surfaces are guarded.

Relationship: this overlaps with SPEC-wf-031 (plugin documentation stays in sync with the
skills), which already names `sdd-help` as a surface kept consistent by the docs-sync
mechanism. Engage should reconcile against wf-031 — either fold this in as a strengthening of
that item (adding the "has help content per skill," not just enumerated, requirement) or as a
distinct item — rather than duplicating it.

Open decisions to settle at target-engage (not pinned here):
- **Fold into wf-031 vs. a new spec item** — avoid two competing requirements.
- **Source of per-skill help** — derived from each `SKILL.md` frontmatter `description`, or
  richer hand-authored help; and whether `sdd-help` is generated from `SKILL.md` like the
  README table or maintained differently.
- **Drift mechanism** — reuse the existing skills-table generator + CI drift check
  (SPEC-wf-031) extended to cover `sdd-help`, or a dedicated check.

Scope note: SDD-plugin meta (`workflow` domain); closely tied to SPEC-wf-031. Exact
decomposition settled during target-engage.
