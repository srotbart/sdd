# Target: Spec test status ‚Äî mapping files, report parsing, and live UI indicators

## Current statement

The Specs screen has no visibility into test coverage or test health per spec item. There is no way to tell from the Hub UI whether a spec item is tested, passing, or failing.

The system needs four cooperating pieces:

**1. Per-spec mapping file** (`.sdd/specs/SPEC-{abbrev}.tests.json`)

A companion JSON file next to each spec declares which test report to read and which test names map to which spec items:

```json
{
  "runner": "vitest",
  "report": "hub/test-results/vitest.json",
  "items": {
    "SPEC-arch-001": ["server runtime is Node.js"],
    "SPEC-arch-005": ["websocket endpoint", "ws client connects"],
    "SPEC-arch-008": ["listens on port 22351"]
  }
}
```

- `runner` ‚Äî `"vitest"` or `"maven"` ‚Äî selects the report parser
- `report` ‚Äî path to the report file (or Surefire directory), relative to workspace root
- `items` ‚Äî spec item ID ‚Üí array of test name substrings; a test matches if its `fullName` contains any listed substring (case-insensitive)

**2. Report parsers**

Both parsers produce a shared internal shape:

```ts
interface TestResult {
  fullName: string;
  status: 'passed' | 'failed' | 'skipped';
}
interface TestReport {
  results: TestResult[];
  runAt: string; // ISO timestamp
}
```

- **Vitest**: run with `--reporter=json --outputFile=<path>`. Parser walks `testResults[*].assertionResults`, reads top-level `startTime` for the timestamp.
- **Maven Surefire**: `report` points to a directory. Parser reads all `TEST-*.xml` files; `fullName` = `classname + " " + name`; failure/error child element = failed; `<testsuite timestamp>` for the timestamp.

**3. Status computation**

Four states per spec item, always set (never absent):

| State | Meaning |
|---|---|
| `passing` | ‚â•1 mapped name found in report; all matched tests passed |
| `failing` | ‚â•1 matched test failed |
| `missing` | No entry in `items` for this spec item, or no matched test names in report |
| `not-run` | Report file does not exist |

`SpecItem` gains:

```ts
testStatus: {
  status: 'passing' | 'failing' | 'missing' | 'not-run';
  lastRun?: string; // ISO timestamp; absent only for 'not-run'
}
```

The backend computes this for every spec item on every `/workspaces/:id/specs` call. Items in a spec domain with no companion `.tests.json` file all receive `{ status: 'not-run' }`.

**4. Extended watcher**

On startup and whenever any `.sdd/specs/*.tests.json` file changes, `startWatcher` collects all `report` paths declared across all mapping files and adds them to the chokidar watch set alongside `.sdd/`. When a report file changes (i.e. tests just ran), the existing `broadcastSddChanged` path fires, the UI refetches specs, and test status updates live.

**5. Frontend ‚Äî `TestStatusDot` component**

A small colored circle with an always-visible dimmed timestamp rendered on the spec item ID line, between the spec ID and the existing `StatusPill`:

```
SPEC-arch-001  ‚óè  2026-05-18 14:32  active   aliases: none
```

Colors:
- `passing` ‚Üí green
- `failing` ‚Üí red
- `missing` ‚Üí amber/yellow
- `not-run` ‚Üí gray

Every spec item always renders the dot. No opt-in, no hiding.
