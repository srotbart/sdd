# Target: "Next steps" skill that recommends prioritized actions for the user to choose

## Current statement

A new SDD skill that, when invoked, surveys the current `.sdd/` state and proposes a set
of candidate next steps for the user to pick from — going beyond `session-start`'s single
footer suggestion to an actionable, ranked menu.

1. **Recommends, doesn't just report.** Where `session-start` renders state and ends with
   one next-action line, this skill produces several candidate actions, each with a short
   rationale for why it's worth doing now.

2. **Ranked by priority, recommendation, and size.** Each candidate carries three signals:
   - **Priority** — urgency from SDD state (e.g. targets awaiting user, stale audits,
     blocked/pending work items, uncovered spec items).
   - **Recommendation** — the agent's judgement of what is most valuable to do next,
     including cross-artifact reasoning (e.g. "engage TGT-096 and TGT-097 together").
   - **Size** — an estimate of the effort/scope of the action so the user can weigh quick
     wins against larger pieces.

3. **The user chooses.** The skill presents the ranked options interactively and lets the
   user select one (this is a required behavior, not optional); the chosen action then
   leads into the relevant existing skill (e.g. `target-engage`, `spec-audit`,
   `gap-to-work-items`, `work-item-close`, `spec-test`).

Open decisions to settle at target-engage (not pinned here):
- **Skill name/invocation** (e.g. `/sdd:next` or `/sdd:recommend`).
- **Relationship to `session-start`** — reuse its state scan vs. duplicate it; whether
  this is a separate skill or an interactive mode of `session-start`. Avoid divergent
  state-reading logic.
- **How "size" is estimated** — heuristic from artifact type/counts, T-shirt sizing, or
  agent judgement; and how "priority" maps onto the existing `session-start` urgency order.
- **How many candidates** to surface and how the selection hands off to / launches the
  downstream skill.

Scope note: SDD-plugin meta (`workflow` domain) — a new skill over existing `.sdd/` state.
Exact decomposition settled during target-engage.
