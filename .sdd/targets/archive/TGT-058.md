# Target: WS snapshot/update must deliver enriched WorkspaceData; UiMessage union must include all 5 types; WorkspaceCounts must match client shape

## Current statement

Three related gaps around the WebSocket data pipeline:

**1. Snapshot/update messages deliver bare Workspace, not enriched WorkspaceData — runtime crash.**
`ws-ui.ts:33` — `buildSnapshot()` calls `getAllWorkspaces()` which returns bare `Workspace[]` (id, name, path, description, created_at only). The client casts `m.workspaces as WorkspaceData[]` (App.tsx:278) and immediately accesses `ws.counts.targetsAwaitingUser` (App.tsx:473) — crashing with "Cannot read properties of undefined" whenever a snapshot or update message arrives. The WebSocket path completely overwrites the correctly-enriched REST data with un-enriched data. Fix: `buildSnapshot()` and `broadcastUpdate()` must call `getWorkspacesEnriched()` instead of `getAllWorkspaces()`.

**2. Server WorkspaceCounts has only 4 fields; client type expects 12.**
`server/index.ts:81-86` computes `{ targetsAwaitingUser, openGaps, workInProgress, workBlocked }`. The client `WorkspaceCounts` type (`types.ts:37-50`) also expects `targetsAwaitingAgent`, `targetsReady`, `targetsDraft`, `specs`, `specItems`, `staleAuditDomains`, `workPending`, `workDoneToday`. The server must be extended to compute and return all 12 fields, or the client type must be reduced to the 4 the server actually produces.

**3. `UiMessage` union in `ws-ui.ts` omits `agent-registered` and `activity` types.**
`ws-ui.ts:26` — `export type UiMessage = StateSnapshot | UpdateMessage | SddChangedMessage`. The two message types broadcast via `broadcastRaw()` (`agent-registered` and `activity`) have no TypeScript interface. Both must be added to the union with formal schemas consistent with SPEC-arch-025.
