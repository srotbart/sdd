# Target: URL reflects selected workspace, tab, and item so refresh restores the same view

## Current statement
Navigation state (active workspace, active tab, selected item within a tab) is held only in React state and is lost on refresh. The URL must encode this state so that reloading the page restores the exact same view.

**URL shape** — query-string params on the existing single-page app (no server-side routing required):
- `?w=<workspaceId>` — the active workspace ID, or absent when Hub dashboard is active
- `&v=<tab>` — the active tab/screen: `session`, `targets`, `specs`, `gaps`, `work-items`, `settings`, `plugin-reference`
- `&id=<itemId>` — the selected item within the tab (e.g. `TGT-019`, `GAP-scr-001`); omitted when no item is selected

Examples: `/?w=ws-1&v=targets&id=TGT-019`, `/?w=ws-1&v=gaps`, `/?` (Hub dashboard)

**On mount** — `App.tsx` reads the URL params before defaulting to localStorage / first workspace. URL params take precedence over `localStorage`. After applying URL state, the URL is considered consumed (no ongoing sync with localStorage beyond what already exists).

**On navigation** — whenever the active workspace, tab, or selected item changes, `App.tsx` updates the URL using `history.replaceState` (no new history entry per click; only explicit user navigations via the sidenav or item selection push a new entry). The `localStorage` key `hub.activeWorkspaceId` continues to be written on workspace change as before.

**Fallbacks** — if a URL-encoded workspace ID is not in the workspace list, fall back to localStorage then first workspace. If the tab is unrecognised, default to `targets`. If the item ID doesn't exist in the current data, open the tab with no item selected.
