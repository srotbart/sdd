# Target: ArtifactList owns filtering â€” derives tabs from items, applies filter internally

## Current statement

The Targets screen and Gaps screen both maintain their own filter state, pre-split items into active/archived arrays, and render their own inline filter bar tabs. This is duplicated logic that should live inside `ArtifactList`.

`ArtifactList` is redesigned to own the full filtering pipeline:

**New API:**
```ts
interface ArtifactListProps<T> {
  items: T[];                          // ALL items â€” unfiltered, unsplit
  filterKey: keyof T;                  // field to group filter tabs by (e.g. "status")
  archivedValues: string[];            // values of filterKey that count as "archived"
  renderRow: (item: T) => ReactNode;
  getKey: (item: T) => string;
  filterLabels?: Record<string, string>; // optional display name overrides per value
}
```

**Internal behaviour:**
1. Derive filter tabs by collecting unique values of `items[filterKey]`, plus a hard-coded "all" tab.
2. Compute per-tab counts from the full `items` array.
3. Maintain `activeFilter` state internally (default: `"all"`).
4. Split items into active (not in `archivedValues`) and archived (in `archivedValues`) after applying the active filter.
5. Render the filter bar above the list â€” a row of tab buttons with label and muted mono count chip, accent treatment on the active tab. Styles live in `ArtifactList.css`.
6. Render filtered active rows (each wrapped in `.artifact-list-active-row`) then the collapsible archived divider and archived rows.

**Callers are simplified:**
- Pass a flat `items` array â€” no pre-splitting into active/archived
- Pass `filterKey="status"` and `archivedValues={['accepted','archived','closed','deferred','done','abandoned']}`
- Remove all inline filter state, filter bar JSX, and active/archived split logic from Targets.tsx and Gaps.tsx

The visual treatment of the filter bar matches the Targets screen's current filter bar exactly (same font size, tab style, active accent, count chip). Existing consumers that do not need filtering pass `filterKey` pointing to a constant field or omit it â€” the component remains backwards-compatible by making `filterKey` optional and rendering no filter bar when absent.
