# Target: Floating terminal button that opens the Claude session in an xterm.js terminal

## Current statement

A **floating terminal button** sits at the bottom of the page. Pressing it opens a terminal
panel that shows the **Claude session**, rendered with **xterm.js** in the client and backed
by **node-pty** on the Hub server.

1. **Floating button** — always-available control anchored at the bottom of the page; toggles
   the terminal panel open/closed.
2. **xterm.js terminal** — the panel renders an interactive terminal via xterm.js.
3. **node-pty backend** — the Hub server uses node-pty to provide the pseudo-terminal that the
   terminal attaches to, streamed to the client (typically over a WebSocket).
4. **Shows the Claude session** — the terminal surfaces the Claude session.

(xterm.js, node-pty, and the bottom-floating button are the user's chosen tech/placement and
are pinned.)

Resolved decisions (see Dialog 2026-06-05 — Agent) — decomposed into two spec items:
- **"The Claude session"** — spawn a `claude` PTY via node-pty with `cwd` = the active
  workspace path (attaching to an already-running host session is not feasible); fall back to
  the platform default shell if `claude` is not on PATH.
- **Transport** — a WebSocket on the existing http server, following the `ws-ui` / `ws-agent`
  attach pattern; closing the panel closes the socket and kills the PTY.
- **Security** — the Hub already binds `127.0.0.1` only with no auth and full workspace FS
  access; the terminal WS inherits exactly that boundary. Not remotely reachable, so arbitrary
  command execution within localhost is accepted and recorded explicitly in SPEC-arch-043.
- **Lifecycle & UX** — one PTY per panel; xterm fit addon ↔ PTY resize; dispose xterm + kill
  PTY on close.

Decomposition (split across domains, matching TGT-112):
- **SPEC-ui-022** (ui-layout) — floating button + xterm.js terminal panel.
- **SPEC-arch-043** (architecture) — node-pty PTY backend over a localhost WebSocket.

### 2026-06-05 — Agent
Tech and placement are pinned (xterm.js, node-pty, bottom-floating button). The open decisions —
including the security one flagged as "settle before implementation" — collapse against the
existing server architecture:

- **Security boundary already exists.** `hub/server/index.ts` binds `127.0.0.1` only (port
  22351), runs no auth, already attaches WebSocket servers (`ws-ui`, `ws-agent`), and already
  has full workspace filesystem access plus agent-spawning. A PTY WebSocket on the same server
  inherits that exact, not-remotely-reachable boundary. I recorded in SPEC-arch-043 that
  arbitrary execution within localhost is accepted — consistent with what the Hub can already do.
  (If you want a stricter posture — token gate or opt-in flag — say so and I'll revise arch-043.)
- **"The Claude session" → spawn a `claude` PTY** (cwd = active workspace path). Reattaching to
  an already-running host session is not feasible, so the terminal starts a `claude` session in
  the workspace; falls back to the default shell if `claude` isn't on PATH.
- **Transport →** WebSocket via the existing `attach*WebSocketServer` pattern; close → kill PTY.
- **Lifecycle →** one PTY per panel, xterm fit addon ↔ PTY resize, dispose on close.

Split across **SPEC-ui-022** (ui-layout: button + panel) and **SPEC-arch-043** (architecture:
node-pty + WS), per the scope note and the TGT-112 precedent. Flipping to `accepted`.
