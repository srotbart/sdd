# Target: Sidenav shows active-artifact counts next to every artifact item

## Current statement

The sidenav shows the number of active (non-archived) artifacts next to each artifact-bearing
nav item. The count mechanism already exists (`tabCounts` → `sidenav-nav-count`, governed by
SPEC-scr-018, "counts reflect non-archived items only"), but today it only covers **targets,
gaps, work items, and specs**. Extend it so every artifact item shows its count — adding
**issues** (`ISS-*`), **improvements** (`IMP-*`), and the document artifacts **projections**,
**designs**, and **standards**.

The three non-artifact tabs (`session`, `activity`, `settings`) intentionally stay countless.

Active semantics per type (resolved — see Dialog 2026-06-05 — Agent):
- **Targets / gaps / work items / specs** — unchanged (SPEC-scr-018).
- **Issues** — active = status `open` or `in-progress` (exclude `resolved`, `wont-fix`).
- **Improvements** — active = status `open` or `in-progress` (exclude `done`, `wont-do`).
- **Projections / designs / standards** — status-less documents; count = number of documents
  present (excluding any `archive/`), analogous to how specs count all domains.

A zero count renders no badge (existing behavior: the badge shows only when count > 0); counts
update reactively as workspace data changes.

Scope: extends the existing sidenav counting (`ui-screens`, SPEC-scr-018); the `tabCounts` prop
is assembled in `App.tsx` and consumed by `Sidenav`. App already holds `liveIssues` /
`liveImprovements`; counts for projections/designs/standards require App to source those
document counts (their screens fetch `/workspaces/:id/{projections,designs,standards}` today).

### 2026-06-05 — Agent
Intent is clear; resolving the flagged decisions from evidence in the code.

- **Active semantics — issues/improvements** derive from their status unions in `types.ts`:
  `IssueStatus = open | resolved | wont-fix | in-progress` → active = `open`/`in-progress`;
  `ImprovementStatus = open | done | wont-do | in-progress` → active = `open`/`in-progress`.
  This mirrors SPEC-scr-018's "exclude terminal statuses" rule.
- **Active semantics — projections/designs/standards** are status-less documents, so the count
  is the number of documents present (excluding any `archive/`), the same way specs count all
  domains. The `Projections`/`Standards` screens already fetch `/workspaces/:id/...` arrays, so
  `length` is the count; App will need to source these (it does not hold them today).
- **Non-artifact tabs stay countless** — `session`, `activity`, `settings` carry no count.
- **New spec item, not a rewrite of SPEC-scr-018.** 018 owns the non-archived rule for
  targets/specs/gaps/work-items and has passing tests; extending coverage to five more tabs is
  additive, so it lands as a new item that references 018 rather than bumping 018's version and
  invalidating its tests.
