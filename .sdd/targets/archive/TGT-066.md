# Target: Clarify SPEC-uic-005 single-source-of-truth exemption for non-ArtifactStatus domains

## Current statement

SPEC-uic-005 states: "No other component may define its own status-to-color or status-to-label mapping; all status rendering must go through `StatusPill`."

`TestStatusDot.tsx` defines its own `STATUS_COLORS: Record<TestStatusKind, string>` mapping `passing/failing/missing/not-run` to raw hex values. This is a technically correct violation of the categorical spec language, but `TestStatusKind` is a completely separate domain from `ArtifactStatus` — the four test status values are not artifact lifecycle states and have no representation in `StatusPill`'s `STATUS_MAP`.

The spec must be clarified: SPEC-uic-005's single-source-of-truth rule applies to `ArtifactStatus` values only. Components that render a distinct, non-artifact status domain (such as `TestStatusKind` in `TestStatusDot`, or hypothetical future domains like `BuildStatus`) may define their own color mappings. The rule prevents fragmented rendering of artifact lifecycle states (`open`, `closed`, `in-progress`, etc.) — it does not prohibit all status-like color mappings in the entire application.

Update SPEC-uic-005 to add: "This rule applies to `ArtifactStatus` values. Components that render a distinct non-artifact status domain with no overlap with `ArtifactStatus` may define their own color mappings within their own component file."
