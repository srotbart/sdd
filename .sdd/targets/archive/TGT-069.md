# Target: watcher.test.ts 4 failures — investigate and fix watcher callback regression

## Current statement

`hub/server/watcher.test.ts` has 4 failing tests. All assert that `onChange` or `onSpecsChanged` callback counts are greater than 0, but the actual count is 0 — meaning the watcher is not invoking its callbacks when files change. This is a potential regression introduced when `startWatcher` was extended to support test report files (SPEC-arch-023) and later when the watcher swap logic was added (SPEC-arch-034).

The sandbox network restriction is not the cause (these are filesystem tests, not network tests). The failures suggest the chokidar event pipeline or debounce timer is not firing correctly in the test environment.

Investigate and fix: read `watcher.test.ts` to understand what the tests expect, read the current `watcher.ts` implementation, identify why callbacks are not being invoked, and fix the watcher or the tests as appropriate. If the watcher logic is correct but the tests need updating to account for the extended `startWatcher` signature (e.g. the new `onSpecsChanged` parameter), update the tests. If the watcher itself is broken, fix it.

Note: the 14 server test failures attributed to `ws-agent`, `ws-ui`, and `ws-ui2` are sandbox network restrictions (`EPERM: listen`) and are NOT code regressions — do not fix those.
