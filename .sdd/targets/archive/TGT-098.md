# Target: Plugin documentation stays in sync with the skills

## Current statement

The plugin's user-facing documentation must always reflect the actual set of skills and
their descriptions — drift between the two should be impossible to ship unnoticed.

1. **Skills are the source of truth.** Each skill's `SKILL.md` frontmatter (`name`,
   `description`) is canonical. Plugin-level documentation that enumerates or describes
   skills derives from / must agree with it. The relevant doc surfaces are at least: the
   root `README.md` "## Skills" table and "Getting Started" examples, the `plugin.json`
   `description`, the `sdd-help` skill content (which also lists skills), and any
   `references/` docs.

2. **Drift is detected, not trusted to memory.** There must be a mechanism that flags
   divergence: a skill directory with no documentation row, a documented skill that no
   longer exists, or a description that no longer matches the `SKILL.md`. Today the README
   "## Skills" table already drifts — it lists 9 skills while `plugin/skills/` contains 11
   (missing at least `explain`, `install-statusline`, and `spawn-sdd-worker`). That drift
   is the kind of thing this target makes impossible to leave unnoticed.

3. **Docs are kept current as part of the flow.** When a skill is added, removed,
   renamed, or its description changes, the plugin documentation is regenerated or
   verified — ideally automatically (generated and/or checked) rather than by hand.

Open decisions to settle at target-engage (not pinned here):
- **Exact scope** of "plugin documentation" (README skills table + getting-started,
  `plugin.json` description, `references/*`, `marketplace.json`) and of "skills
  documentation" (`SKILL.md` frontmatter; the `sdd-help` enumeration).
- **Mechanism** — generate vs. verify: a script that regenerates the README table from
  `SKILL.md` frontmatter, a pre-commit hook / CI check that fails on drift, a dedicated
  skill, or a combination. One-way (docs derived from skills) vs. bidirectional.
- **Coverage of `sdd-help`** — it is a second skill-enumerating surface and must be kept
  consistent by the same mechanism.
- **Adjacent template tweak (salvaged from dropped TGT-099)** — while this work touches
  plugin docs/templates, optionally make the *"Open decisions to settle at engage"*
  section a standard part of the target template, plus a one-line reminder that a
  target's "what" must be testable. Small, related; fold in only if cheap, otherwise
  leave out of scope.

Scope note: SDD-plugin meta (`workflow` domain) — about the plugin's own documentation
maintenance, not the Hub UI. Exact decomposition settled during target-engage.
