# Target: Comprehensive spec-test coverage view in the Specs UI

## Current statement
The Hub Specs view should present a comprehensive picture of each spec item's test coverage, not just a single status dot. Concretely:

1. **Per-item test detail** — In the spec-item detail view, render the item's linked tests individually: each test's name, its own status (passing / failing / missing), and last-run time, sourced from the parsed test report — not only from the markdown `**Tests:**` block.

2. **Domain coverage summary** — On the existing Specs screen (a header / summary strip, not a separate screen), show a per-domain rollup: covered / total plus counts of passing, failing, missing, and not-run items, so coverage is scannable at a glance across domains.

3. **Skipped vs uncovered are distinguished** — Items deliberately skipped (no meaningful code boundary) are surfaced distinctly from items that simply have no tests yet, with the skip reason visible, so a gray "not-run" item is not mistaken for an intentionally-untestable one. A skip is recorded via a convention in the spec item file — a `**Tests:** skipped — <reason>` line — which the parser and UI read.

4. **Live status is wired, not gray** — The mechanism that computes per-item test status (per-domain mapping file `SPEC-{abbrev}.tests.json` + generated Vitest/Surefire report, per SPEC-arch-019/020/022) must actually be populated for this repo, so statuses reflect real test runs instead of defaulting to "not-run". Today no mapping files or report exist, so every item renders gray.

5. **Coverage tooling keeps the wiring in sync** — The `/sdd:spec-test` skill that adds coverage must also create/maintain the per-domain `SPEC-{abbrev}.tests.json` mapping files (and the report-path wiring), so that linking a `**Tests:**` block and making the UI show live status are a single consistent step rather than two disconnected ones.

Scope note: primarily `ui-screens` (Specs screen + spec-item detail) and `ui-components` (status rendering), with `workflow` impact on the `sdd:spec-test` skill and a dependency on the existing `architecture` test-mapping/report mechanism (SPEC-arch-019/020/022, currently unexercised for this repo). Exact spec-item decomposition settled during reconciliation.
