# Target: App.tsx crashes when WebSocket snapshot workspace has no counts field

## Current statement

`App.tsx:408` — `alertCount: ws.counts.targetsAwaitingUser` — crashes with `TypeError: Cannot read properties of undefined (reading 'targetsAwaitingUser')` when a workspace in a WebSocket `snapshot` or `update` message has no `counts` field. This occurs in the `workspaces.map(ws => ({ ..., alertCount: ws.counts.targetsAwaitingUser }))` call.

The root cause is that the client casts WebSocket message payloads as `WorkspaceData[]` but does not guard against the case where a workspace in the payload has `counts: undefined`. While SPEC-arch-036 mandates that `buildSnapshot()` and `broadcastUpdate()` use `getWorkspacesEnriched()` (now implemented), defensive coding at the client is still required.

Fix: replace `ws.counts.targetsAwaitingUser` with `ws.counts?.targetsAwaitingUser ?? 0` (and equivalently for any other `ws.counts.*` access). This prevents the crash when the server sends a workspace without counts — which can happen during startup, partial hydration, or in test fixtures.

Also fix: `App.test.tsx` snapshot test fixture should include a `counts` field to match the `WorkspaceData` contract, making the test representative of real server output.
