# Target: Artifact IDs inside markdown content auto-render as ArtifactPeeker links

## Current statement

Artifact IDs that appear **inside rendered markdown content** (e.g. `SPEC-wf-025` written in
a spec invariant or a target's statement) should automatically render as clickable links
that open the artifact in the `ArtifactPeeker` — the same behavior `ArtifactIdLink` already
gives to structured ID cells (SPEC-uic-012), now extended to prose. This happens
automatically in the shared `Markdown` component (SPEC-uic-013), so it works everywhere
markdown is rendered, with no per-call-site wiring.

1. **Automatic in markdown** — IDs matching `TGT-`/`SPEC-`/`GAP-`/`WI-`/`ISS-`/`IMP-`
   patterns in markdown text become peeker links.
2. **Reuses existing machinery** — links open via the existing `ArtifactIdLink` /
   `ArtifactPeeker`; unknown/unloaded IDs fall back to plain text.
3. **Centralized** — implemented once in the shared `Markdown` component; every screen that
   renders markdown gets it.

Open decisions to settle at target-engage (not pinned here):
- Implementation: a remark plugin that rewrites matching text into link nodes (with an
  `artifact:` sentinel URL) + a `components.a` override mapping those to `ArtifactIdLink`.
- The exact ID regex (cover `TGT-001`, `SPEC-uic-001`, etc.; avoid over-matching).
- Skip IDs inside code spans/blocks (don't linkify code samples) and inside existing
  markdown links.

Scope note: extends the shared `Markdown` component (`ui-components`); reuses
`ArtifactIdLink`/`ArtifactPeeker`. Exact decomposition settled during target-engage.
