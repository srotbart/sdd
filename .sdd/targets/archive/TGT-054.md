# Target: TargetStatus type missing 'archived' — resolve spec vs type inconsistency

## Current statement

SPEC-scr-018 says the sidenav target count should exclude both `accepted` and `archived` targets. The `TargetStatus` type (`types.ts:16`) defines only: `'awaiting-user' | 'awaiting-agent' | 'ready' | 'draft' | 'accepted'` — `'archived'` is absent.

The Targets screen (SPEC-scr-003) shows an "archived" filter tab. The SDD target lifecycle uses `archived` as the terminal directory state (targets move to `targets/archive/`). The backend parser returns these targets with `status: 'accepted'` (since `accepted` is the last status before archiving). The client's `mapApiTarget` at `App.tsx:106-125` has no `'archived'` case — it falls back to `'draft'` for any unrecognized status.

The gap: either (a) the type should include `'archived'` as a valid terminal status that the parser sets when reading from `targets/archive/`, or (b) the spec should clarify that `accepted` IS the archived state and remove the word "archived" from SPEC-scr-018.

Evidence suggests option (a) is correct: the Targets screen filter tab is labeled "archived" (SPEC-scr-003), and reading targets from both `targets/` and `targets/archive/` without distinguishing them in the status field makes archive display impossible. The parser should set `status: 'archived'` for targets read from `targets/archive/`, and the type system should include it.
