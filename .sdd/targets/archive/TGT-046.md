# Target: WebSocket message schemas formally specced and client consumes snapshot, update, and reconnect

## Current statement

SPEC-arch-005 says the server sends "a full state snapshot" on connection and "typed update events" subsequently, but defines no message schemas. Three concrete message types exist in the code with no spec coverage: `snapshot` (`ws-ui.ts:7-12`), `update` (`ws-ui.ts:13-18`), and `agent-registered` (`ws-agent.ts:76`). The `activity` event emitted from `ws-agent.ts:92-97` is also unspecced.

The client (`App.tsx:289-319`) currently handles only `type: "sdd-changed"` and drops all other message types with an early return at line 294. This means the real-time state snapshot the server sends on every new connection is silently discarded — the client never processes live workspace or agent state from WebSocket, only from the periodic REST fetches.

Additionally, there is no WebSocket reconnection logic anywhere in `App.tsx`. When `ws.onclose` fires, `hubConnected` is set to false and the connection is abandoned permanently. No spec item requires reconnection behavior.

Three spec items are needed:

1. **Formal schemas for all server→client WebSocket message types**: `snapshot`, `update`, `sdd-changed` (already in SPEC-arch-018), `agent-registered`, and `activity` — each with their field names and semantics.

2. **Client handles `snapshot` and `update` messages**: on `snapshot`, the client should update live workspace and agent state from the WebSocket payload rather than requiring a REST refetch. On `update`, same.

3. **WebSocket reconnection**: when the connection drops, the client should attempt to reconnect with exponential backoff (e.g., 1s, 2s, 4s, max 30s). The spec should define the reconnection strategy and what the UI shows while disconnected.
