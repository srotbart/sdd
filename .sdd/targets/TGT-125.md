# Target: Auto-start the hub on session start, with a single-instance guarantee

## Current statement

When Claude Code starts, it should **connect to the hub via MCP automatically**, and if the
hub is not running, **start it** — without the user ever manually traversing into the versioned
plugin cache directory (`~/.claude/plugins/.../sdd/hub`). No matter how many Claude Code sessions
are open, there must be **exactly one hub instance running** at a time.

**Chosen approach** (settling the prior open decisions):
- A **`SessionStart` hook** runs a launcher script on every Claude Code start.
- The script does a **liveness probe** on the hub's fixed address (`127.0.0.1:22351`). If the hub
  answers, it does nothing (just lets MCP connect). If not, it **starts the hub** (detached) after
  acquiring an **atomic global/file lock** used as a semaphore, so two sessions starting at the
  same instant cannot both spawn a hub. The lock is released once the hub is up (or on failure).
- The **fixed port is itself the steady-state singleton lock**, and port-switching stays disabled:
  if `22351` is already bound, a second hub must **fail rather than pick a new port**.

**What the code already gives us (grounding, verified):**
- `hub/server/index.ts` already binds a **fixed** `HOST=127.0.0.1` / `PORT=22351` and, on
  `EADDRINUSE`, prints "already running… only one instance may run at a time" and `process.exit(1)`
  — it does **not** fall back to another port. So the steady-state single-instance guarantee
  already exists at the server; this target adds the *auto-start* and *race-free start* around it.
- The launcher's file lock is therefore a **fast-path race guard** for the start window; the
  server's `EADDRINUSE`-exit is the **backstop** that guarantees singleton even if the lock is
  bypassed or stale.
- The **Vite dev client** (`hub/client/vite.config.ts`, port `22400`) has **no `strictPort`**, so
  it *will* auto-increment when the port is taken — this is the one place "disable port switching"
  is not yet enforced and should be closed for dev parity.

It must:
- **Connect-or-start on session start** via the `SessionStart` hook + launcher; never require the
  user to know the plugin cache path (resolve the hub location version-resiliently).
- **Guarantee one instance** under concurrent session starts (file-lock fast path + the existing
  port-bind/`EADDRINUSE` backstop); later sessions just connect to the running hub.
- **Keep port-switching disabled** end to end: server already exits on a taken port; add
  `strictPort` (or equivalent) so the client does the same.
- **Be resilient**: detect and reclaim a **stale lock** after a hub crash; surface a clear message
  on start failure rather than wedging or silently retrying forever.
- **Be cross-platform**: the hook script, the lock primitive, and detached spawn must work on both
  Windows (PowerShell) and POSIX (cf. hub Node 22.x runtime and the lockfile registry-pinning gotcha).

Open decisions to settle at target-engage (not pinned here):
- **Lock primitive** — OS file lock / lockfile-with-PID / atomic `mkdir` / named mutex; and exactly
  how stale-lock reclaim decides the holder is dead (PID liveness vs. port probe vs. TTL).
- **Liveness definition** — plain TCP/port probe on `22351`, an HTTP health endpoint, or the MCP
  handshake itself as the readiness signal.
- **Lifecycle / ownership** — does the hub keep running after the launching session exits (shared
  daemon) or get torn down; who, if anyone, stops it.
- **Hook wiring** — script lives in the plugin and is referenced via `${CLAUDE_PLUGIN_ROOT}`; how it
  is registered (plugin-provided hook vs. user `settings.json`) so install is zero-touch.
- **Failure UX** — what the user sees if autostart fails (visible error vs. silent, and whether MCP
  connection failure alone should trigger a restart attempt).
