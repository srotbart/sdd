# Target: Dark mode support

## Current statement
The Hub UI should support a dark mode in addition to the current light ("paper") theme.

1. **Token-based theming** — the CSS custom properties in `hub/client/src/styles/tokens.css` (surfaces `--paper*`, ink `--ink*`, hairlines `--hair*`, accent `--accent*`, status `--st-*`) are the single source of truth for colour. A dark theme supplies an alternative value set for these tokens, applied at the root (e.g. a `[data-theme="dark"]` attribute or a class on the app shell), so individual components need no per-component dark styles — they already consume the tokens.

2. **Theme toggle** — a user-facing control to switch between light and dark (and optionally a "system" option). Most naturally lives in the Settings screen and/or the header.

3. **Persistence & system preference** — the selected theme persists across reloads (localStorage); a "system" mode follows the OS `prefers-color-scheme`.

4. **Legibility on dark surfaces** — status colours (the passing/failing/missing/skipped dots and `--st-*`) and the accent remain legible and reasonably contrasty on dark backgrounds; dark-theme token values are tuned where the light values would be too dark or low-contrast.

5. **No flash of light theme** — the initial theme is applied before first paint to avoid a light flash on load.

Scope note: primarily `ui-layout` (global theming + shell). Touches `ui-components` (any component using hardcoded colours instead of tokens must migrate to tokens for the swap to work) and the Settings screen (`ui-screens`) for the toggle control. Exact spec-item decomposition settled during target-engage.
