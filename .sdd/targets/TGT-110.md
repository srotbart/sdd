# Target: Markdown rendering supports richer content (tables and other GFM)

## Current statement

The Hub renders artifact markdown with support for richer content — at minimum **GFM
tables**, which the current `react-markdown` setup does not render (markdown tables in
specs, targets, etc. currently show as raw text). Ideally other common GFM features render
too (strikethrough, task lists, autolinks). The renderer is either upgraded to a richer one
or extended (e.g. a GFM plugin), and a shared markdown-rendering component is used across the
screens so the capability is consistent everywhere it appears.

1. **Tables render** — markdown tables in artifact content display as tables, not raw pipes.
2. **Richer GFM content** — common GitHub-Flavored-Markdown features (strikethrough, task
   lists, autolinks) render.
3. **Consistent everywhere** — one shared markdown renderer is used across the artifact
   surfaces. (Today `react-markdown` is invoked inline in ~6 places: `SpecItemList`,
   `SpecItemDetail`, `Targets`, `Designs`, `Projections`, and the new artifact screens.)

Open decisions to settle at target-engage (not pinned here):
- **Extend vs. replace** — add `remark-gfm` to the existing `react-markdown`, or switch to a
  different renderer.
- **Which GFM features** to enable (tables only, or the full common set).
- Whether to introduce a shared `<Markdown>` component and refactor the existing call sites
  onto it (vs. just adding the plugin at each site).
- Sanitization/safety of any newly-enabled syntax (raw HTML stays disabled unless decided).

Scope note: Hub UI rendering, primarily `ui-components` (a shared renderer) consumed across
screens. Exact decomposition settled during target-engage.
