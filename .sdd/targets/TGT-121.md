# Target: Define and update a project's north star

## Current statement

A project using SDD should be able to **define and update its north star** — a persistent,
single statement of the project's overarching prime goal / guiding vision that frames every
target and spec beneath it. The north star is the "why" the rest of the SDD artifacts serve;
individual targets are specific intents, but the north star is the durable, project-level
intent they ladder up to.

It must be:
- **Definable** — the user sets the north star for the project (the source of truth, like
  standards: user-authored, the agent does not invent it).
- **Updatable** — the user can revise it as the project's direction evolves, with the change
  tracked (history / dated revisions), not silently overwritten.
- **Surfaced** — the agent receives the north star as context (e.g. at session-start, the
  same proactive-delivery channel used for the operating contract and orientation) so its
  work stays aligned to it.

Open decisions to settle at target-engage (not pinned here):
- **Where it lives** — a dedicated `.sdd/` artifact (e.g. `.sdd/north-star.md`) vs. a field
  in an existing place; single-file vs. structured.
- **Relationship to existing concepts** — how it differs from / complements the persona
  concept (CLAUDE.md `# Persona`) and from individual targets; whether targets should be able
  to reference/justify against it.
- **Surfacing mechanism** — session-start orientation, sdd-help, and/or a Hub view; how it is
  delivered without duplication-drift (cf. SPEC-wf-028/029 standards-delivery pattern).
- **Update/versioning** — how revisions are recorded (dated dialog vs. version hash) and
  whether prior north stars are archived.
- **Scope** — per-project only, or also a notion of sub-goals.
