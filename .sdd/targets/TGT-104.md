# Target: Test indicators shown on each spec artifact list item

## Current statement

The test-status indicators should appear **as part of each spec artifact list item** — the
per-domain rows in the spec list (the left sidebar in `hub/client/src/screens/Specs.tsx`,
lines ~114–128) — so coverage is visible per artifact directly in the list.

1. **Indicators on the list item.** Today a spec artifact row shows only name, `id`,
   version, and item count; the test status lives only in the separate top coverage strip
   (`Specs.tsx` ~95–109). Each spec artifact list item should carry its own test-status
   indicators so the user sees coverage per artifact while scanning the list.

2. **Reuse existing test-status vocabulary.** The same status semantics already used
   elsewhere (the `TestStatusDot` component and the passing / failing / missing / not-run /
   skipped counts in `computeDomainCoverage`) drive the indicator — no new status model.

Open decisions to settle at target-engage (not pinned here):
- **Indicator form on the row** — a single aggregate `TestStatusDot`, a compact
  passing/failing/missing distribution, or a coverage fraction (e.g. `17/21`).
- **Strip vs. row** — whether moving indicators onto the rows replaces the standalone
  top coverage strip, or supplements it.
- **Layout fit** — how the indicator sits within the compact sidebar row without crowding
  the name/version/count.

Scope note: the Specs screen spec list (`ui-screens`); reuses the existing
`TestStatusDot` / coverage computation. Exact decomposition settled during target-engage.
