# Target: Floating terminal button that opens the Claude session in an xterm.js terminal

## Current statement

A **floating terminal button** sits at the bottom of the page. Pressing it opens a terminal
panel that shows the **Claude session**, rendered with **xterm.js** in the client and backed
by **node-pty** on the Hub server.

1. **Floating button** — always-available control anchored at the bottom of the page; toggles
   the terminal panel open/closed.
2. **xterm.js terminal** — the panel renders an interactive terminal via xterm.js.
3. **node-pty backend** — the Hub server uses node-pty to provide the pseudo-terminal that the
   terminal attaches to, streamed to the client (typically over a WebSocket).
4. **Shows the Claude session** — the terminal surfaces the Claude session.

(xterm.js, node-pty, and the bottom-floating button are the user's chosen tech/placement and
are pinned.)

Open decisions to settle at target-engage (not pinned here):
- **"The Claude session"** — attach to an existing/host Claude session vs. spawn a new `claude`
  process via node-pty; and, in this multi-workspace Hub, which workspace's session.
- **Transport** — WebSocket wiring between xterm.js and the node-pty process; reconnection.
- **Security (important)** — a web-served PTY is a live shell. Decide the trust boundary:
  local-only binding, auth, and whether arbitrary command execution is acceptable. This must be
  settled before implementation.
- **Lifecycle & UX** — one PTY per panel/tab, terminal resize ↔ pty resize (xterm fit addon),
  cleanup on close, panel size/expand behavior.

Scope note: spans the app shell UI (`ui-layout`: floating button + terminal panel) and the Hub
server (`architecture`: node-pty + WebSocket); engage will likely split into items across both
domains. Exact decomposition settled during target-engage.
