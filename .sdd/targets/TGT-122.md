# Target: A tracked registry of drift-prone surfaces in the SDD project

## Current statement

The SDD project has several **derived surfaces** that restate information whose source of
truth lives elsewhere, and so can silently fall out of sync ("drift"). This target introduces
a single **tracked registry of drift-prone surfaces**: each entry names a surface, its
authoritative source, and how the surface is kept in agreement (a generator and/or a check).
The point is that drift becomes systematically detectable and fixable instead of discovered
by accident — as just happened with the `projection-comments` skill missing from the README
and sdd-help (ISS-wf-004).

Known drift-prone surfaces to seed the registry (non-exhaustive):
- **Plugin reference / README "## Skills" table** — source: each `SKILL.md` frontmatter;
  generator `gen-skills-table.js`, check `check-skills-drift.js` (SPEC-wf-031).
- **sdd-help "## All Skills"** — same source; generator `gen-sdd-help-skills.js` (SPEC-wf-033).
- **`references/schemas.md` and the per-artifact guides** — source: `references/artifacts/*.md`;
  check `check-artifact-guides.js` (SPEC-wf-023); skills must reference the guides, not restate.
- **session-start operating contract / orientation** — derived from the artifact guides and
  the pipeline model (SPEC-wf-024/030); must not diverge from sdd-help / the guides.
- **sdd-init scaffolding** — the directory structure / templates it creates must match the
  artifacts the rest of the pipeline expects.
- **plugin.json description**, and any other place that enumerates or describes skills.

It must:
- **Enumerate** each drift-prone surface with its source of truth and its check/generator.
- **Be the one place** to look to answer "what can drift here, and how is it caught?"
- Ideally tie each entry to an automated check so the registry is enforceable (e.g. surfaced
  through `lint-check.sh` / CI), not just documentation.

Open decisions to settle at target-engage (not pinned here):
- **Form** — a new `.sdd/` artifact type, a doc under `references/`, or a spec item that lists
  the surfaces; whether it is data (machine-readable) or prose.
- **Enforcement** — does each entry require an automated check, and do they roll up into one
  `check-drift` command / CI gate?
- **Relationship to existing specs** — SPEC-wf-023/024/030/031/033 already each govern a
  surface; is the registry a meta-index over them, or does it supersede/centralise them?
- **Generality** — SDD-project-specific only, or a general capability for any SDD project.
