# Target: Organic knowledge base for the worker (graph + semantic search)

## Current statement

During implementation the sdd-worker repeatedly re-discovers the same things — how
infrastructure works, where to look for a given concern, how to retrieve a piece of
data — by searching the codebase from scratch every time. This target introduces a
persistent, self-populating knowledge base the worker consults instead of re-searching.

1. **A queryable knowledge store** — a graph database with semantic (vector) search.
   The worker can ask it natural-language questions ("how is X configured?", "where
   does Y live?") and get back stored knowledge plus the related nodes linked to it.
   Graph structure (entities + relationships, e.g. concept → file, task → procedure,
   subsystem → how-to) so related knowledge is connected, not just a flat doc pile.

2. **Organic / self-populating** — the knowledge base grows as the worker works. When
   the worker learns something — a repeated data-retrieval result, a how-to, a
   "where to look for things" pointer, a piece of infrastructure understanding — it
   writes that back so the next task doesn't repeat the search. It accretes project
   working-knowledge over time rather than being authored up front.

3. **Content types** — at minimum: (a) cached/repeated **data retrieval** results,
   (b) **how-to** procedural knowledge, (c) **location** knowledge ("where to find X"),
   and (d) general **memory/knowledge** about what's needed to work on the project
   (infrastructure especially).

4. **Retrieval-first workflow** — the intended loop is: query the knowledge base first;
   on a miss (or stale hit) fall back to searching the codebase; then write the finding
   back so the next lookup is a hit. The win is fewer redundant codebase searches.

Open decisions to settle at target-engage (not pinned here):
- **Storage tech** — embedded vs server-based graph DB; which engine; how vector/semantic
  search is provided (embeddings model + index). The `cognify` MCP server already
  connected to this session is a candidate backend worth evaluating before building one.
- **Access surface** — how the worker reads/writes it: MCP tool, a skill, or a hub
  service. Whether it is a general sdd-worker capability or scoped to this hub project.
- **Scope & provenance** — per-project vs global knowledge; how entries are attributed
  and dated.
- **Staleness/invalidation** — how cached retrieval and location knowledge are
  invalidated when the underlying code changes, so the worker doesn't act on stale hits.

Scope note: this opens a genuinely new domain (`knowledge-base`) — worker/execution
infrastructure rather than the Hub UI. Spec-item decomposition settled during
target-engage.
