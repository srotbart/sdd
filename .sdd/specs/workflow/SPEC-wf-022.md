# SPEC-wf-022 — sdd:spec-test maintains test mapping files and live report wiring

## Invariant

When the `/sdd:spec-test` skill adds coverage to a domain, it does more than write a markdown `**Tests:**` block into each spec item: it also creates and maintains the per-domain test mapping file `.sdd/specs/{domain}/SPEC-{abbrev}.tests.json` (declaring `runner`, `report` path, and the item-to-test `items` map) and ensures the report path is produced by the project's test run, so the Hub computes live `testStatus` instead of defaulting every item to `not-run`. Linking a test and lighting up the UI status are a single consistent step. The skill also records a deliberate skip using the `**Tests:** skipped — <reason>` convention (per SPEC-scr-047) rather than leaving skipped items indistinguishable from uncovered ones.

## Acceptance criteria

- Running `/sdd:spec-test {domain}` writes/updates `.sdd/specs/{domain}/SPEC-{abbrev}.tests.json` with `runner`, `report`, and an `items` map keyed by spec item ID
- The `items` map entries match the test `fullName`s the skill linked in each item's `**Tests:**` block (so `computeTestStatus` resolves them)
- The skill ensures the declared `report` path is generated by the test runner (e.g. a Vitest JSON reporter), so status reflects a real run
- Items the skill skips are written with a `**Tests:** skipped — <reason>` line instead of being left blank
- After the skill runs, the Hub Specs UI shows live per-item status for the covered items rather than `not-run`
